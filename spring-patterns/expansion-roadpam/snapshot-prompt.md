당신의 역할:
- 이 프로젝트를 학습용 클론 코딩에 사용하기 위해,
  1) CLONE-GUIDE.md 와 design/ 디렉터리를 기준으로 패치 단위의 이력을 정리하고,
  2) 각 패치 시점의 전체 소스코드를 snapshot/ 디렉터리에 스냅샷으로 생성하며,
  3) 최종적으로 문서와 실제 소스코드 상태의 정합성을 점검하는 것이다.

입력으로 가정하는 현재 상태:
- 프로젝트 루트에 다음이 존재한다고 가정한다.
  - CLONE-GUIDE.md
  - design/ 디렉터리 (버전/마이너 단위의 설계 문서들이 일부 존재할 수 있음)
  - 실제 동작하는 최신 소스코드 (이 최신 상태가 CLONE-GUIDE 상의 마지막 단계에 해당한다고 가정)
- snapshot/ 디렉터리는 없을 수도 있고, 있어도 비어 있거나 임시 상태일 수 있다.

추가 전제:
- 현재 레포지토리의 "최신 버전" 태그(예: v1.2.3)는 CLONE-GUIDE.md 마지막 항목을 기준으로 한다.
- 사람이 필요시 다음과 같이 명시해 줄 수 있다.
  - 예: "현재 기준 버전: v1.2.3"

당신이 해야 할 일 (순서대로):

1. 전체 흐름 분석
   - CLONE-GUIDE.md 를 읽고, 프로젝트가 어떤 "버전 / 마이너 / 패치" 단위로 진행되는지 전부 파악하라.
   - design/ 디렉터리의 설계 문서를 모두 읽고, 어떤 버전/단계에 대해 설계가 이미 존재하는지 정리하라.
   - CLONE-GUIDE.md 에는 패치 단위까지 정의되어 있지만, design/ 에는 마이너 단위로만 작성되어 있을 수 있다.
     - 이 경우, CLONE-GUIDE.md 의 패치 단위 내용을 기준으로 design/ 쪽 설계 문서를 패치 단위로 세분화·보강해야 한다.

2. design/ 정렬 및 보강
   - CLONE-GUIDE.md 에 정의된 각 "패치 단위"에 대해,
     - 해당 패치 시점에서의 시스템 상태, 주요 변경점, 인터페이스, 데이터 구조, 예외 케이스 등을 정리한 설계 문서를 design/ 아래에 준비하라.
   - 이미 design/ 에 마이너 단위 설계가 있다면,
     - 그 마이너 버전 내부를 패치 단위로 나누는 형태로 섹션을 쪼개거나,
     - 필요하다면 패치 단위 설계 파일을 추가 생성하라.
   - 이때, design/ 안의 파일명·디렉터리 구조·문서 형식은 "기존에 사용 중인 패턴"을 그대로 유지하면서 확장하라.
     - 새로운 파일을 만들 때도 기존 네이밍 규칙을 따라라.

3. snapshot/ 디렉터리 준비
   - 프로젝트 루트에 snapshot/ 디렉터리가 없다면 생성하라.
   - snapshot/ 아래에는 "패치 버전"을 식별할 수 있는 하위 디렉터리를 만든다.
     - 예시: snapshot/v1.0.0, snapshot/v1.0.1, snapshot/v1.1.0-1, ...
     - 실제 이름 규칙은 CLONE-GUIDE.md 에 사용된 버전/패치 표기를 따라라.

4. 패치 단위 스냅샷 생성 (최신 버전 바로 직전까지만)
   - CLONE-GUIDE.md 상의 가장 초기 단계부터 시작하여, 최종 단계(현재 소스코드 상태에 해당하는 마지막 버전) 직전까지 순서대로 진행하라.
   - 각 패치 단계에 대해 다음을 수행하라.
     1) 직전 패치 상태의 코드에서, CLONE-GUIDE.md 와 design/ 설계에 따라 해당 패치에서 요구하는 변경을 적용해라.
     2) 이 패치 적용 후의 상태가 "빌드 가능 + 테스트 통과"인 일관된 상태가 되도록 코드를 정리하라.
        - 필요한 경우, 테스트 코드를 새로 작성하거나 기존 테스트를 패치 내용에 맞게 수정해도 된다.
     3) snapshot/<패치버전>/ 디렉터리에 이 시점의 전체 프로젝트 소스코드를 복사해라.
        - 부분 코드가 아니라, 이 패치 시점에서의 전체 프로젝트 구조가 들어 있어야 한다.
     4) snapshot/<패치버전>/ 내부에서 실제로 빌드와 테스트를 수행해 보고, 성공하는지 확인하라.
        - 실패 시, 원인을 해결하고 다시 테스트해서 통과한 상태의 코드만 스냅샷으로 남겨라.

5. 정합성 검증
   - 다음 네 가지가 서로 모순 없이 일치하는지 검증하라.
     1) CLONE-GUIDE.md 에 서술된 단계별 설명
     2) design/ 설계 문서(패치 단위로 보강된 상태)
     3) snapshot/ 에 존재하는 각 패치별 코드 스냅샷
     4) 최종 현재 소스코드 상태
   - 특히,
     - CLONE-GUIDE.md 마지막 단계의 설명과 실제 현재 소스코드가 기능·인터페이스·데이터 구조 측면에서 정합적인지 확인하라.
     - 정합성이 깨지는 부분이 있다면, 어느 파일/어느 버전 설명과 상충되는지 구체적으로 리포트하라.
   - 정합성 문제를 발견해도, 근거가 불명확한 상태에서 코드나 문서를 함부로 뜯어고치지 말고, 우선 "어디가 어긋났는지"를 명확히 문서화하라.

제약 사항:
- 빌드와 테스트를 위한 명령 외에, 프로젝트 상태를 불필요하게 변경하는 터미널 명령은 사용하지 마라.
- 버전 이력 자체를 조작하거나, 외부 저장소/서비스와 통신하는 작업은 수행하지 마라.
- CLONE-GUIDE.md 와 design/ 의 기존 형식, 톤, 네이밍 규칙을 최대한 그대로 유지하면서 확장하라.
- 모든 스냅샷 코드는 "바로 빌드와 테스트를 실행할 수 있는 상태"여야 한다.

출력:
- 처리 과정에서 어떤 패치 버전들에 대해 어떤 스냅샷 디렉터리를 생성했는지 목록을 요약하라.
- design/ 에 새로 추가하거나 변경한 문서 목록과, 각 문서가 어떤 패치/버전을 다루는지 정리해서 보고하라.
- 발견한 정합성 문제(있다면)를 버전/파일 단위로 요약해서 보고하라.
