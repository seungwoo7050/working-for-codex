[역할]
너는 이미 존재하는 프로젝트에 "추가 구현 요구사항"을 반영하면서,
문서(CLONE-GUIDE.md / design/)와 snapshot/ 디렉터리를 모두 일관되게 갱신하는 작업을 담당한다.
코드는 항상 빌드/테스트 가능한 상태의 단계들로만 진행해야 한다.

[전제]
- 현재 작업 디렉터리는 하나의 프로젝트 루트라고 가정한다.
- 프로젝트에는 대략 다음과 같은 것들이 존재한다:
  - CLONE-GUIDE.md
    - 기능/변경사항을 "패치 단위"로 나눈 클론 코딩용 진행 가이드
  - design/
    - 버전/패치 단위의 설계 문서들
  - snapshot/
    - 특정 버전/패치 시점의 스냅샷 코드가 저장되는 디렉터리
  - 실제 코드 디렉터리
  - 테스트 관련 설정/스크립트 파일들
- "현재 코드 상태"를 하나의 기준 버전으로 본다.
  - 필요하다면 CLONE-GUIDE.md / design/ 에서 사용 중인 규칙을 따라 버전명을 정하고, 그 버전명을 사용한다.

[입력으로 주어지는 것]
- 추가 구현 요구사항 문서 경로 목록
  - expansion-roadpam/v1.6.md
- 이 문서들에는 "마이너 단위" 수준의 요구사항이 적혀있다고 가정한다.
  - 너는 이를 "패치 단위"까지 쪼개서 처리해야 한다.

[최종 목표]
이미 존재하는 프로젝트에 대해 다음을 수행하는 것이 목표다:

1) 현재 기준 버전의 코드 상태를 snapshot/ 에 스냅샷으로 안전하게 고정한다.
2) 추가 구현 요구사항 문서를 "패치 단위"로 쪼개어 CLONE-GUIDE.md 에 반영한다.
3) 새로 반영된 CLONE-GUIDE.md 를 기준으로 design/ 하위 설계 문서를 작성/갱신한다.
4) 각 패치 단위로 실제 코드 구현 + 빌드 + 테스트를 수행한다.
5) 테스트가 통과하여 해당 버전에 대한 구현이 끝났다면 snapshot/ 을 갱신한다.
6) 제시한 전체 구현이 마무리되면, CLONE-GUIDE.md / design/ / 실제 코드 / snapshot/ 의 정합성을 검증하고 결과를 요약해서 보고한다.

## 0. 프로젝트 상태 파악

0-1. 다음을 확인한다:
  - CLONE-GUIDE.md 존재 여부
  - design/ 디렉터리 존재 여부 및 하위 구조
  - snapshot/ 디렉터리 존재 여부 및 기존 스냅샷 네이밍 규칙
  - 빌드/테스트에 사용될 설정/스크립트 (예: package.json, Makefile, build.gradle, README 등)

0-2. 필수 요소가 비어 있거나 구조적으로 심각하게 깨져 있다면:
  - 어떤 파일/디렉터리가 부족한지, 왜 진행이 어려운지 서술하고
  - 임의로 구조를 새로 만드는 대신, 진행을 중단하고 보고만 한다.

0-3. 이후 작업에서 사용할 "현재 기준 버전"을 다음 규칙으로 정리한다:
  - CLONE-GUIDE.md / design/ 에 이미 버전명이 있다면, 가장 최신 버전명을 현재 기준 버전으로 사용한다.
  - 없다면, 기존 규칙을 참고하여 새로운 버전명을 제안하고,
    - 예: v1.2.0, v2025-01-01 등
  - 어떤 이름을 기준 버전으로 사용하는지 명시한다.

## 1. 현재 기준 버전 스냅샷 고정

1-1. snapshot/ 디렉터리를 살펴보고, 현재 기준 버전에 대응하는 스냅샷이 이미 존재하는지 확인한다.
  - 기존 스냅샷의 디렉터리/파일 구조, 네이밍 규칙을 파악한다.

1-2. 이미 동일한 기준 버전의 스냅샷이 존재한다면:
  - 스냅샷을 변경하지 않고, "이미 존재함"을 기록만 한다.

1-3. 존재하지 않는다면:
  - snapshot/ 하위에 현재 기준 버전용 스냅샷을 추가한다.
  - 추가 시 규칙:
    - 기존 스냅샷 구조와 최대한 동일한 디렉터리/파일 구조를 따른다.
    - 빌드와 테스트를 재현하는 데 필요한 최소한의 파일 집합을 포함한다.
  - 어떤 경로에 무엇을 복사/생성했는지, 재현 방법은 무엇인지 명확히 기록한다.

1-4. 만약 기존 규칙이 "최종 상태는 별도 스냅샷을 만들지 않는다"와 같이 명시되어 있다면,
  - 그 규칙을 존중하고, 필요 최소한의 스냅샷만 추가한다.

## 2. 추가 구현 요구사항을 패치 단위로 분해 + CLONE-GUIDE.md 반영

2-1. 입력으로 주어진 추가 구현 요구사항 문서들을 모두 읽고, 내용과 범위를 파악한다.
  - 각 문서별로 어떤 기능/변경사항이 포함되어 있는지 목록화한다.

2-2. 각 요구사항을 "패치 단위"로 쪼갠다.
  - 한 패치는 적용 후에 빌드/테스트가 가능한 안정 상태가 되도록 작게 나눈다.
  - 서로 의존성이 있는 패치들은 순서를 분명히 한다.
  - 여러 문서에서 나온 항목이라도, 하나의 일관된 패치 순서로 재정렬한다.

2-3. CLONE-GUIDE.md 를 수정/추가한다.
  - 기존의 구조/서식/톤을 그대로 유지한다.
  - 새로 정의한 각 패치에 대해:
    - 패치 ID 또는 이름
    - 간단한 목적/요약
    - 해당 패치가 참조하는 요구사항 문서 경로 및 섹션(있다면)
    - 구현 순서 상의 위치(어느 버전 뒤에 오는지)
  - 기존 내용과 섞일 때 혼동되지 않도록, 섹션/버전 구분을 명확히 한다.

2-4. CLONE-GUIDE.md 변경 결과로,
  - "패치 단위의 전체 진행 순서"가 한 눈에 보이도록 정리한다.

## 3. CLONE-GUIDE 기반으로 design/ 작성/갱신

3-1. 새로 추가된 패치들에 대해, design/ 하위에 대응하는 설계 문서를 생성하거나 기존 문서를 갱신한다.
  - design/ 의 기존 디렉터리 구조/파일명 규칙을 유지한다.
  - 예: design/v1.3/patch-01-xxx.md 와 같은 규칙이 있다면 그대로 따른다.

3-2. 각 패치 설계 문서에는 최소한 다음 내용을 포함한다:
  - 패치 ID / 이름, 관련 버전
  - 요구사항 요약 (어떤 문서의 어느 부분에서 왔는지)
  - 변경되는 인터페이스, 데이터 구조, DB 스키마, 외부 의존성 등
  - 주요 처리 흐름(간단한 시퀀스/플로우 설명)
  - 예상되는 사이드 이펙트와 주의사항
  - 테스트 관점에서 중요한 시나리오(나중에 테스트 코드와 연결될 수 있게)

3-3. 이미 존재하는 design/ 문서와 충돌/중복되는 부분이 있다면:
  - 어떤 문서가 구버전/신버전 역할을 하는지 정리하고,
  - 어디를 기준으로 삼을지 분명히 기록한 뒤, 최종 설계 기준을 하나로 정리한다.

## 4. 패치 단위 구현 + 빌드 + 테스트

4-1. CLONE-GUIDE.md 에 정리된 순서대로 패치를 하나씩 처리한다.
  - "여러 패치의 구현을 동시에 섞어서" 진행하지 않는다.
  - 항상 "한 패치 -> 빌드/테스트 그린 -> 다음 패치" 순서를 지킨다.

4-2. 각 패치에 대해 실제 코드 변경을 하기 전에:
  - 이 패치에서 변경될 파일/모듈 범위를 설계 문서와 함께 재확인한다.
  - 예상 빌드/테스트 영향 범위를 파악한다.

4-3. 코드 구현을 진행한다.
  - design/ 에 정의된 설계와 CLONE-GUIDE.md 설명을 우선 기준으로 삼는다.
  - 필요 시, 요구사항 문서를 다시 참조해 의도와 일치하는지 확인한다.

4-4. 빌드와 테스트를 수행한다.
  - 프로젝트에서 이미 사용 중인 빌드/테스트 명령을 우선 사용한다.
    - 예: package.json 의 scripts, Makefile, README 에 적힌 명령 등
  - 빌드/테스트 명령이 여러 개라면:
    - 어떤 명령들을 실행했는지 명시하고,
    - 최소한 핵심 테스트 스위트는 항상 실행한다.

4-5. 빌드/테스트가 실패하면:
  - 실패 원인을 분석하고, 패치 범위 안에서 원인을 해결한다.
  - "임시로 테스트를 끄거나" 하는 식으로 문제를 숨기지 않는다.
  - 테스트가 모두 통과하기 전에는 다음 패치로 넘어가지 않는다.

4-6. 해당 패치가 안정 상태에 도달하면:
  - CLONE-GUIDE.md 또는 design/ 에,
    - 구현 중에 변경되거나 추가로 발견된 사항(예: 제약, 예외 케이스)을 정리해 반영한다.

## 5. snapshot/ 갱신

5-1. 패치 단위 또는 특정 버전 단위로, snapshot/ 을 갱신한다.
  - 기존 snapshot/ 구조/네이밍 규칙을 우선적으로 따른다.
  - 새로 추가된 패치/버전이 기존 스냅샷 체계에서 어떤 위치에 대응하는지 분명히 한다.

5-2. 각 스냅샷에 대해 다음을 명확히 한다:
  - 이 스냅샷이 어떤 패치/버전 상태를 나타내는지 (CLONE-GUIDE 상의 위치)
  - 어떤 빌드/테스트 명령으로 재현 가능한지
  - 포함된 디렉터리/파일 목록(요약 수준)

5-3. 기존 규칙이 "최종 완료 상태는 스냅샷을 따로 만들지 않는다"라면:
  - 그 규칙을 유지하면서도, 중간 단계들이 충분히 커버되도록 스냅샷 포인트를 선정한다.

## 6. 정합성 검증 및 결과 보고

6-1. 정합성 체크 목록:
  - CLONE-GUIDE.md 에 정의된 모든 새 패치가:
    - design/ 어딘가에서 설계로 구체화되어 있는지
    - 실제 코드 상의 변경과 일치하는지
  - snapshot/ 의 각 항목이:
    - CLONE-GUIDE.md 상의 어떤 단계/버전에 대응하는지 명확한지
    - 실제로 그 상태로 빌드/테스트가 가능한지 (필요 시 샘플링하여 검증)
  - 추가 구현 요구사항 문서에 있는 항목이:
    - 누락 없이 패치/설계/구현/스냅샷에 반영되었는지

6-2. 무작위로 몇 개의 패치를 선택하여:
  - 요구사항 문서 → CLONE-GUIDE → design/ → 실제 코드 → snapshot/ 을 따라가며,
    - 내용이 논리적으로 연결되는지 확인하고,
    - 불일치가 있으면 어디에서/왜 생겼는지 기록한다.

6-3. 전체 구현 및 갱신 작업이 끝나면, 다음 내용을 포함하는 요약 보고를 만든다:
  - 추가된/수정된 패치 목록 (ID, 요약, 관련 요구사항 문서)
  - 새로 추가되거나 변경된 design/ 문서 목록
  - 새로 추가되거나 변경된 snapshot/ 항목 목록 및 대응 버전/패치
  - 빌드/테스트 수행 내역과 현 시점의 상태 (전부 통과/부분 실패 등)
  - 남아 있는 이슈나 TODO, 애매한 부분(있다면)

6-4. 마지막으로:
  - "이 프로젝트를 클론 코딩 관점에서 어떻게 따라가면 되는지" 관점에서,
    - 어떤 순서로 CLONE-GUIDE.md / design/ / snapshot/ / 실제 코드를 읽고 따라가면 좋은지
    - 학습자가 주의해야 할 포인트가 무엇인지 간단히 정리한다.
